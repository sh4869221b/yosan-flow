## Current controller ordering (observed source fact)

`runClientEffect()` starts `Effect.runPromise()` without returning or awaiting it (`src/lib/dashboard/client-effect.ts`). In `submitDayEntryEffect()` (`src/lib/dashboard/day-entry-controller-state.svelte.ts`), the successful POST path currently:

1. sets the returned period summary;
2. updates `selectedRow` when `selectedDate` is non-null;
3. waits for `historyController.loadHistoryEffect(selectedDate)`;
4. calls `closeDayEntry()`;
5. clears `modalSaving`.

Therefore a save button click can complete at the browser-action level before history refresh and modal close finish. The failing specs immediately reopen or repeat entry operations without explicitly pairing the click with the successful POST response and modal-hidden state. This is supporting evidence for a stale modal/history completion race, not yet root-cause proof.

## Planned RED/GREEN gates

1. **Day-entry generation race RED:** instrument or route-delay the successful day-add/history sequence; perform save, reopen a newer modal session, release the old history completion, and require the current implementation to demonstrate that old completion can close or overwrite the newer session. If 20 barrier-verified attempts cannot reproduce it, reject the production controller guard and leave controller source unchanged.
2. **Day-entry generation race GREEN:** if RED is proven, bind completion to the originating modal generation so an old completion cannot mutate/close a newer session. The same deterministic scenario must pass, with public API and timeout/retry policy unchanged.
3. **Mutation wait GREEN:** for every successful day-entry add, pair the click with the exact period/day POST 2xx response and then require `day-entry-modal` hidden before the next action. Negative validation/503 cases keep direct error assertions.
4. **JST boundary RED/GREEN:** independently reproduce test/server clock divergence at the declared boundary. Do not implement a shared clock unless the RED exists; require both boundary cases to pass after any clock fix.
5. **Reset/stale-state gate:** prove the detector sees deliberately seeded state before reset and zero state after reset for the declared repetition count.
6. **Final stress gate:** four successful-add specs at 400/400, negative-adjustment grep at 100/100, full repository CI-parity commands, real desktop/mobile browser flows, and a remote run with retained failure diagnostics.
