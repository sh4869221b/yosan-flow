# Yosan Flow 設計書 v1.1

## 1. 文書情報

### 1.1 システム名称

- 論理名: **Yosan Flow**
- repository 名: **`yosan-flow`**

### 1.2 文書の目的

本書は、Yosan Flow の業務要件、ドメイン設計、アーキテクチャ、API、データ構造、運用前提を整理し、実装計画へ接続するための設計書である。

### 1.3 名称の位置づけ

Yosan Flow は、月予算を基準に、日々の使用額入力に応じて残額を再計算し、今日以降へ再按分した推奨日次予算を提示する個人用アプリケーションである。

---

## 2. 目的

Yosan Flow は、月予算を設定し、日ごとの使用額入力に応じて残額を再計算し、**今日以降に使える推奨日次予算**を提示する個人用アプリケーションである。

本アプリは以下を満たす。

- Cloudflare Workers 上で動作する
- フロントエンドは SvelteKit を使用する
- 永続データは D1 に保存する
- アプリ全体は Cloudflare Zero Trust / Access で保護する

---

## 3. スコープ

### 3.1 対象

- 月予算の設定と変更
- 日ごとの使用額の追加と上書き
- 未来日の予定支出登録
- 今日以降の推奨日次予算の表示
- 対象日の履歴表示

### 3.2 非対象

- 複数ユーザー対応
- カテゴリ別管理
- 収入管理
- 履歴の取消・削除
- 銀行連携や外部家計簿連携
- 月次確定や締め処理のワークフロー

## 4. 確定仕様

### 4.1 基本条件

- 通貨は円固定
- 利用者は個人 1 名
- 日付基準は JST 固定
- 画面表示単位は 1 か月
- 新しい月の予算初期値は前月予算を引き継ぐ
- 月途中の予算変更を許可する
- 履歴の取消機能は不要
- アプリは Cloudflare Zero Trust で保護する

### 4.2 日次入力

- 日ごとの使用額は **追加** と **上書き** の両方に対応する
- 未来日入力を許可する
- 未来日入力は UI 上 **予定支出** として表示する
- 日付到来後は通常の使用額として扱う

### 4.3 再按分

- 再按分表示は **今日以降のみ**
- 月予算変更時は、過去使用分を保持したまま今日以降へ再按分する
- 予算超過時は以下とする
  - 推奨日次予算を 0 円表示
  - 超過額を別表示する

### 4.4 端数処理

- 再按分の端数は、**今日から順に 1 円ずつ配分する**
- 例: 残額 100 円、残り 3 日なら以下とする
  - 1 日目 34 円
  - 2 日目 33 円
  - 3 日目 33 円

### 4.5 過去分表示

- 日別一覧は今日以降のみ表示する
- ただし、**今月ここまでの使用額** はヘッダで必ず表示する

### 4.6 初回月

- 前月データが存在しない場合は、予算未設定状態で開始する
- 初回アクセス時に月予算入力を促す
- 初回表示時の月レコード自動作成は行わない

## 5. アーキテクチャ

### 5.1 構成

- UI: SvelteKit
- サーバ処理: SvelteKit server routes
- 実行基盤: Cloudflare Workers
- 永続化: Cloudflare D1
- 認証・保護: Cloudflare Access

### 5.2 採用方針

Yosan Flow の主責務は、月予算、日次集計、履歴、再計算である。そのため、リアルタイム協調専用の構成より、SQL ベースで整合性を持ちやすい D1 中心構成を採用する。

SvelteKit 側は `adapter-cloudflare` を使用し、Cloudflare 側は Workers と D1 を binding で接続する。アプリ URL への到達は Access policy により制御する。

### 5.3 実装境界

- ブラウザは表示と入力を担当する
- SvelteKit server routes は入力検証、JST 基準日判定、集計取得、再按分計算結果の返却を担当する
- D1 は月予算、日次集計、履歴の永続化を担当する
- 推奨日次予算は保存せず、取得時に都度計算する

### 5.4 時刻基準

- システム上の「今日」は JST の日付で判定する
- Cloudflare Workers 実行環境の UTC 時刻をそのまま業務日付に使わず、JST に変換して扱う

## 7. 業務ルール

### 7.1 追加

その日の合計使用額へ入力金額を加算する。

### 7.2 上書き

その日の合計使用額を入力値で置き換える。

### 7.3 予定支出

未来日入力は予定支出として表示する。ただし計算上は、**月内で既に見込まれている支出** として残額に反映する。

### 7.4 予定支出の当日化

予定支出専用の別データ構造は持たない。JST の現在日付と対象日付の比較により、表示ラベルだけを切り替える。

- 今日より後: 予定支出
- 今日以前: 通常の使用額

表示切替は、**レスポンス生成時点の JST 日付** を基準に毎回判定する。バッチ更新は行わず、画面再取得時に自然反映させる。

### 7.5 月予算未設定時

- 月予算未設定の月では日次入力を完了できない
- UI は予算設定モーダルまたは入力導線を優先表示する
- API は `budget_yen` 未設定時に日次更新要求を拒否する
- 前月予算候補がある場合は、初期入力値として提示してよい

### 7.6 月跨ぎ制約

- 日次登録対象日は URL の月と一致しなければならない
- 他月の日付は登録・更新対象として扱わない

## 8. 再按分ロジック

### 8.1 基本計算

対象月について以下を定義する。

- 月予算 = `M`
- 月内登録済み使用額合計 = `U`
- 月残額 = `R = M - U`

ここで `U` は、対象月の **実績分 + 未来日の予定支出分** を含む月内登録総額とする。

### 8.2 今日の推奨日次予算

- 今日から月末までの残り日数 = `D`
- `R >= 0` の場合
  - 基本額 = `R ÷ D` の整数商
  - 余り = `R mod D`
  - 今日から順に余り分だけ 1 円上乗せする
- `R < 0` の場合
  - 今日の推奨日次予算 = 0 円
  - 超過額 = `abs(R)`

`D` は **今日を含む** 日数とする。

### 8.3 表示対象

再按分結果の一覧は **今日以降だけ** 表示する。

### 8.4 過去日修正

過去日を追加または上書きした場合も、以下を行う。

- 月累計使用額を更新
- 月残額を再計算
- 今日以降の推奨日次予算だけを再表示する

### 8.5 月予算変更

月途中で予算変更した場合は、以下を行う。

- 過去使用額は維持
- 新しい月予算を基準に残額を再計算
- 今日以降へ再按分する

### 8.6 月末最終日

- 対象日が月末の場合、残り日数 `D` は 1 とする
- そのため推奨日次予算は残額そのもの、または超過時 0 円となる

### 8.7 当月外アクセス

- 過去月・未来月の画面表示でも、再按分表示対象は「その閲覧時点の JST 基準日と対象月の関係」により決まる
- 対象月が現在月でない場合は、今日以降一覧の意味が曖昧になるため、初版では**現在月のみを主要表示対象**とする
- 他月表示を許可する場合は、再按分の基準日を別途定義する

## 10. API 設計

### 10.1 共通方針

- レスポンスは JSON とする
- 金額はすべて整数の円で返す
- サーバ側で JST 基準日を評価する
- 更新系 API は正常終了時に、更新後の月集計サマリを返す

### 10.2 月 API

#### `GET /api/months/:yyyy-mm`

用途:

- 対象月の表示用データ取得
- 未作成月に対する初期表示情報の返却

返却内容:

- 月予算
- 月状態
- 引継ぎ有無
- 引継ぎ元月
- 今月ここまでの使用額
- 今月の総使用予定額
- 月残額
- 超過額
- 今日の推奨日次予算
- 今日以降一覧
- 初期候補予算

注意:

- `GET` は副作用を持たない
- 月レコード未作成時も、UI 表示に必要な初期状態を返してよい

想定レスポンス例:

```json
{
  "yearMonth": "2026-04",
  "monthStatus": "ready",
  "budgetYen": 120000,
  "budgetStatus": "set",
  "initializedFromPreviousMonth": true,
  "carriedFromYearMonth": "2026-03",
  "suggestedInitialBudgetYen": 120000,
  "spentToDateYen": 42000,
  "plannedTotalYen": 58000,
  "remainingYen": 62000,
  "overspentYen": 0,
  "todayRecommendedYen": 2067,
  "daysRemaining": 30,
  "dailyRows": [
    {
      "date": "2026-04-18",
      "label": "today",
      "usedYen": 0,
      "recommendedYen": 2067
    }
  ]
}
```

#### `POST /api/months/:yyyy-mm/initialize`

用途:

- 未作成月の明示的初期化
- 前月予算候補を用いた当月作成

想定リクエスト例:

```json
{
  "budgetYen": 120000
}
```

#### `PUT /api/months/:yyyy-mm/budget`

用途:

- 月予算の新規設定
- 月予算の変更

想定リクエスト例:

```json
{
  "budgetYen": 120000
}
```

#### 月未作成時の扱い

月未作成時は `GET` では作成せず、`POST /initialize` または `PUT /budget` の初回呼び出しで作成する。対象月未存在時、前月予算があればその値を初期候補として返す。

### 10.3 日次 API

#### `POST /api/days/:yyyy-mm-dd/add`

用途:

- 追加操作

想定リクエスト例:

```json
{
  "inputYen": 1500,
  "memo": "昼食"
}
```

#### `PUT /api/days/:yyyy-mm-dd/overwrite`

用途:

- 上書き操作

想定リクエスト例:

```json
{
  "inputYen": 3000,
  "memo": "修正後"
}
```

#### `GET /api/days/:yyyy-mm-dd/history`

用途:

- 対象日の履歴取得

### 10.4 エラーレスポンス

代表的な HTTP ステータスは以下とする。

- `400 Bad Request`: 金額形式不正、日付形式不正
- `404 Not Found`: 対象月または対象日が業務上参照不能
- `409 Conflict`: 月予算未設定のまま日次更新を要求
- `500 Internal Server Error`: 保存失敗、取得失敗

想定レスポンス例:

```json
{
  "error": {
    "code": "BUDGET_NOT_SET",
    "message": "月予算を設定してください。"
  }
}
```

## 11. 状態遷移

### 11.1 月表示

1. 対象月を検索
2. 未存在なら前月を検索
3. 前月があれば予算候補として返す
4. 前月がなければ未設定月として表示情報を返す
5. 月集計と今日以降一覧を返す

### 11.1.1 月初期化

1. `POST /initialize` または `PUT /budget` を受ける
2. 対象月の存在を確認する
3. 未存在なら月レコードを作成する
4. 引継ぎ元があれば記録する
5. 表示用集計を返す

### 11.2 追加

1. 対象日の現在合計取得
2. 入力額を加算
3. 日次集計更新
4. 履歴追加
5. 月集計再計算
6. 今日以降推奨予算再表示

### 11.3 上書き

1. 対象日の現在合計取得
2. 入力値で置換
3. 日次集計更新
4. 履歴追加
5. 月集計再計算
6. 今日以降推奨予算再表示

### 11.4 日付経過

バッチや自動変換処理は持たない。日付比較により、予定支出ラベルが自然に通常表示へ変わる。

### 11.5 競合時の扱い

- 初版では単一ユーザーかつ低頻度操作を前提とする
- ただし更新系 API は、日次集計更新と履歴追加を同一トランザクションで処理する
- これにより、日次合計と履歴の不整合を防ぐ
- 同一月初期化の重複要求に対しては、既存レコードを優先し冪等に扱う

## 12. データ設計

### 12.1 D1 テーブル案

#### `monthly_budgets`

- `year_month TEXT PRIMARY KEY`
- `budget_yen INTEGER NULL`
- `budget_status TEXT NOT NULL`
- `initialized_from_previous_month INTEGER NOT NULL DEFAULT 0`
- `carried_from_year_month TEXT NULL`
- `created_at TEXT NOT NULL`
- `updated_at TEXT NOT NULL`

制約:

- `CHECK(budget_status IN ('unset', 'set'))`

#### `daily_totals`

- `date TEXT PRIMARY KEY`
- `year_month TEXT NOT NULL`
- `total_used_yen INTEGER NOT NULL`
- `updated_at TEXT NOT NULL`

制約:

- `CHECK(total_used_yen >= 0)`

インデックス:

- `INDEX idx_daily_totals_year_month (year_month, date)`

#### `daily_operation_histories`

- `id TEXT PRIMARY KEY`
- `date TEXT NOT NULL`
- `operation_type TEXT NOT NULL`
- `input_yen INTEGER NOT NULL`
- `before_total_yen INTEGER NOT NULL`
- `after_total_yen INTEGER NOT NULL`
- `memo TEXT NULL`
- `created_at TEXT NOT NULL`

制約:

- `CHECK(operation_type IN ('add', 'overwrite'))`
- `CHECK(input_yen >= 0)`
- `CHECK(before_total_yen >= 0)`
- `CHECK(after_total_yen >= 0)`

インデックス:

- `INDEX idx_daily_histories_date_created_at (date, created_at DESC)`

### 12.2 保存方針

- `year_month` は `YYYY-MM` 文字列で保持する
- `date` は `YYYY-MM-DD` 文字列で保持する
- 監査上の意味を持つため、履歴は追記のみとする
- 推奨日次予算は保存しない

### 12.3 月初期化時の扱い

- 対象月未存在時、前月の `budget_yen` が存在すればそれを初期候補として返す
- 前月レコードはあるが `budget_yen` が未設定なら、当月も未設定で開始する
- 初回月では `budget_yen = NULL` を許容する
- `budget_status = 'unset'` の月では日次更新を受け付けない

- 金額は 0 以上
- 金額は整数のみ
- 日付は正しい日付であること
- 対象日は指定月内であること

### 13.2 業務制約

### 13.3 表示制約

- 再取得失敗時は明示的に失敗を表示する

## 14. セキュリティ設計

Yosan Flow は Cloudflare Access policy により保護する。そのため、アプリ内部で複雑なユーザー管理や権限管理を持たない。

### 14.1 方針

- 公開 URL は Access 保護下に置く
- 許可対象は本人の ID のみ
- アプリ内は単一ユーザー前提で処理する

### 14.2 アプリ内前提

- Access を通過したリクエストのみがアプリへ到達する前提とする
- 初版ではアプリ内セッション管理を追加しない
- ただし監査や将来拡張のため、Access 由来ヘッダの受け取り方は実装時に確認する
- `workers.dev` と preview URL を未保護のまま公開しない運用方針を採る

### 14.3 データ保護

- 金額、メモ、履歴を D1 に保存する
- 秘密情報は D1 に保存しない
- `.dev.vars` や Cloudflare シークレット値はリポジトリに含めない

## 15. 非機能設計

### 15.1 保守性

- 日次集計と履歴を分離
- 推奨日次予算は保存せず都度計算
- 予定支出は別種別化せず表示で区別

### 15.2 拡張性

将来的に追加しやすい機能:

- 年単位表示
- カテゴリ別支出
- CSV 出力
- 月比較
- 複数予算枠

### 15.3 規模適合性

Yosan Flow は個人利用の小規模アプリであり、想定データ量・アクセス量ともに軽量である。そのため、Cloudflare Workers + D1 構成で十分成立する。

### 15.4 可観測性

- 初版では高度な監視基盤は必須としない
- ただし API エラー件数と D1 失敗は運用時に確認できるよう、Workers ログで追える構成とする

## 16. デプロイ・運用前提

### 16.1 Cloudflare 構成要素

- Workers
- D1 Database
- Cloudflare Access
- 必要に応じて `wrangler` によるデプロイ設定

### 16.2 環境区分

初版では以下を想定する。

- local: SvelteKit + D1 ローカル開発
- preview: Cloudflare Workers preview + preview 用 D1 + Access 方針明示
- production: Cloudflare Workers + D1 + Access

### 16.3 マイグレーション

- D1 スキーマは migration ファイルで管理する
- テーブル作成、インデックス作成、初期データ不要を前提とする
- 破壊的変更が必要な場合は設計書更新とセットで扱う

### 16.4 デプロイ原則

- local / preview / production で D1 を分離する
- バインド名は全環境で統一し、DB 実体のみ切り替える
- Access 保護対象のホスト名を明示し、未保護 URL を残さない
